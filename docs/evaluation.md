# Evaluation

This document summarizes how quality was evaluated across the retrieval and synthesis pipelines, what the results showed, and how findings drove system changes.

---

## Dataset

A golden evaluation dataset of 264 questions was generated by GPT-4o from actual filing chunks sampled from the corpus.

| Dimension | Values |
|---|---|
| Companies | AAPL, AMZN, GOOG, MSFT, NVDA, TSLA, JPM, JNJ, PFE, XOM, KO, UNH |
| Sections | RISK FACTORS (ITEM 1A), MD&A (ITEM 2), BUSINESS (ITEM 1) |
| Filing types | 10-K and 10-Q |
| Fiscal years | 2023–2025 |

**Matching strategy**: Evaluation hits are determined by metadata match (ticker + filing_type + fiscal_year + section_label), not node ID. LlamaIndex regenerates chunk UUIDs on every index load, making ID-based matching unreliable across runs.

---

## Retrieval Evaluation (`eval/run_retrieval_eval.py`)

### Metrics

- **HitRate@50**: Percentage of queries where the golden section appears in the top-50 retrieved chunks
- **MRR**: Mean Reciprocal Rank — average of 1/rank for the first gold-section chunk
- **Multi-Company Coverage**: Percentage of compound queries where both target companies are represented in top-50 results

### Results

| Approach | HitRate@50 | MRR | Multi-Co Coverage |
|---|---|---|---|
| Vector search baseline | 0.981 | 0.536 | 0.467 |
| + Cross-encoder reranking | 0.898 | 0.475 | 0.733 |
| + Ticker-aware MMR | 0.886 | 0.469 | **0.950** |

### Key Finding: The Diversity Problem

Pure vector search is highly accurate for single-company queries (98.1% HitRate) but fails for multi-company comparative queries — only 46.7% of compound queries had both target companies represented in the results. One company consistently dominated the top-K slots, leaving the other absent from the synthesis context with no signal to the user.

Ticker-aware MMR trades ~9 percentage points of HitRate for a 48 pp gain in multi-company coverage (46.7% → 95.0%). Given that the target query types are comparative, this tradeoff is correct: a slightly lower-ranked but complete context is strictly better than a high-ranked but one-sided context.

### Company-Level Weakness

AMZN (70.8% HitRate, 0.218 MRR) and GOOG/MSFT (~79% HitRate, 0.23–0.34 MRR) are consistent outliers across all retrieval approaches. The likely cause is an embedding surface mismatch between how questions are phrased and the language used in those companies' filings — these companies use more abstract or technical language in their risk and MD&A sections.

### The Single-LLM-Call Constraint and Retrieval

Query decomposition — splitting "compare Apple and Microsoft cloud strategies" into two separate focused retrievals — would have solved the coverage problem cleanly. Each sub-query would retrieve 15–20 high-precision chunks for one company, and the synthesis context would arrive pre-organized.

The constraint prevented this, requiring retrieval-layer compensation via ticker-aware MMR. The result is effective (95% coverage) but the soft cap per ticker is a blunt instrument. For queries spanning five or more companies, some companies may still receive fewer chunks than their relevance warrants.

## Synthesis Evaluation (`eval/run_synthesis_eval.py`)

### Setup

46 questions sampled across four strata:

| Stratum | Description | N |
|---|---|---|
| A | Year-specific 10-Q questions | 12 |
| B | Year-specific 10-K questions | 13 |
| C | Multi-year trend questions | 8 |
| D | Year-agnostic questions (year signal stripped) | 13 |

Full retrieval + synthesis pipeline run per question. GPT-4o-mini used as an independent LLM judge, separate from the synthesis model (GPT-4o).

### Metrics

- **Faithfulness** (0–1): Are the specific claims in the answer supported by the cited excerpts?
- **Relevance** (0–1): Does the answer address what the question asks?
- **NLI score** (0–1): Post-hoc sentence-level entailment score from `cross-encoder/nli-deberta-v3-small`, independent of the LLM
- **Calibration**: System's NLI-based confidence rating (HIGH/MEDIUM/LOW) compared to the judge's faithfulness score
  - HIGH → CORRECT if faithfulness ≥ 0.75, else OVERCONFIDENT
  - MEDIUM → CORRECT if 0.45 ≤ faithfulness < 0.75
  - LOW → CORRECT if faithfulness < 0.45, else UNDERCONFIDENT

### Results

| Metric | Score |
|---|---|
| Mean faithfulness (judge) | 0.735 |
| Mean relevance (judge) | 0.870 |
| Mean NLI score | 0.860 |
| Calibration: CORRECT | 67.4% |
| Calibration: OVERCONFIDENT | 21.7% |
| Calibration: UNDERCONFIDENT | 10.9% |

System ratings: 33 HIGH, 6 MEDIUM, 7 LOW across 46 questions.

### Key Finding: Systematic Overconfidence on Numerical Claims

The most consistent failure pattern: the model rated answers HIGH when the judge scored faithfulness below 0.75. The mechanism — the model recalls a financial figure from training memory, cites a related but vague excerpt, and rates itself HIGH — drove two changes:

1. **Prompt v3 — Pre-Answer Check**: An explicit section before answer generation requires the model to verify that any specific number is *visible* in the excerpt text before stating it, naming the failure pattern ("confabulation").

2. **NLI post-hoc scoring**: Replaced LLM self-assessment with `cross-encoder/nli-deberta-v3-small` scoring each cited claim against its source excerpt independently. This eliminated the circularity of asking the generating model to rate its own confidence. The mean NLI score (0.860) is notably higher than judge faithfulness (0.735) — NLI measures entailment of cited sentences only, while the judge catches fabricated specific numbers regardless of how they're cited.

### Relevance vs. Faithfulness Gap

Mean relevance (0.870) substantially exceeds mean faithfulness (0.735). The system consistently understands and addresses the question but doesn't always ground its answer in the retrieved text. This gap is the clearest signal of the synthesis quality problem: answers are on-topic but not always fully traceable to the provided excerpts.

### Year Alignment

Year-agnostic questions (Stratum D) showed the most faithfulness degradation. When no fiscal year signal is present in the query, retrieval remains fully open, and wrong-year chunks enter the synthesis context. This finding motivated the year-based metadata pre-filter: a heuristic extracts explicit years, relative spans ("last 3 years"), and recency signals from query text and applies them as a pgvector metadata filter before ANN search.

## How Eval Findings Drove System Changes

| Finding | Change |
|---|---|
| Multi-company coverage 46.7% on comparative queries | Cross-encoder reranker + ticker-aware MMR |
| MMR output 45 JPM + 3 AAPL + 2 TSLA for 3-company query | Reranker pool expanded to 200; soft-cap MMR replaced with balanced quota selection |
| Systematic overconfidence on numerical claims | Prompt v3 pre-answer check + confabulation guardrail |
| LLM self-assessment circular and unreliable | DeBERTa NLI post-hoc faithfulness scoring (`nli-deberta-v3-small`) |
| Wrong-year chunks polluting retrieval for year-agnostic queries | Year-based metadata pre-filter extracted from query text |
| Model organizing 50-chunk context inconsistently | Prompt v4 structured reasoning chain |
| Multi-company answers covering only one company | Prompt v5/v6 coverage list, per-company pre-draft, audit checklist |
| Comparison questions missing comparison paragraph | Prompt v6 mandatory Comparison paragraph when comparison language detected |
